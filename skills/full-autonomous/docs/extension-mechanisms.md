## G-4 技能清单快照（P1 — 🔧 可实施）

### 目的
跨会话追踪技能集合的变化，辅助缺口检测和路由校准。

### 实现路径

在 Phase 0 STEP 0.1 加载完 `<available_skills>` 后，追加以下步骤：

```
[√] G-4: 技能清单快照 → 写入 skill_snapshot.md
    格式: | 技能名 | 版本 | 首次发现 |
    路径: ~/.reasonix/skill_snapshot.md
    行为: 追加模式（按 date 追加），每会话一行
```

### 写入格式

```markdown
# 技能快照日志

## 2026-05-24T10:00:00
本次新增: 3 个 | 消失: 1 个 | 总计: 113 个
| 技能 | 首次发现 |
|------|----------|
| new-skill | 2026-05-24 |
| missing-old-skill | ~~2026-05-20~~ （最后出现）|
```

### 集成步骤
1. Phase 0 后追加快照写入
2. 与已有 `scan skills/ directory` 共用扫描结果，不需额外遍历
3. 快照文件到达 200 行时自动轮转（保留最新 100 行）

### 验收标准
- [ ] 新会话启动后有 `skill_snapshot.md` 文件生成
- [ ] 文件中包含本次会话的技能总数和新增/消失对比

## S-3 技能性能反馈回路（P0 — ✅ 已在 Phase 5 实现）

### 目的
每次技能执行后记录性能、结果和错误，供 skill-router 优化下次匹配排序。

### 当前实现
Phase 5 阶段 C 的 `STEP 5C.1` 已要求写入 `skill_performance.json`，格式：

```json
{
  "timestamp": "{ISO时间}",
  "skill": "full-autonomous",
  "action": "{任务描述}",
  "assessment": "success|partial|failure",
  "reason": "{一句话评价}",
  "duration_seconds": {N},
  "skills_used": ["{skill1}"],
  "skills_skipped": ["{skill3}"]
}
```

### 扩展建议
如需更细粒度的技能级别反馈，可在 Phase 3 每个子任务完成后追加轻量级记录：

```
[技能反馈] {skill_name} | {duration}s | {result} | {error_type或无}
```

写入同一个 `skill_performance.json` 文件。

## S-6 自动暂停决策模块（P2 — 📋 设计就绪）

### 目的
当系统检测到持续异常时主动降级，避免浪费 token 和陷入死循环。

### 触发条件（任一满足即触发）

| 条件 | 阈值 | 检测方式 |
|------|------|----------|
| 连续子任务超时 | 连续 3 个子任务耗时 > 预估 × 2 | Phase 3 STEP 3.5 循环计数器 |
| 技能缺口率过高 | 无法匹配 skill 的任务占比 > 30% | Phase 0 缺口检测 |
| 同任务反复失败 | 同一任务失败 ≥3 次且原因不同 | ralph 循环计数 |
| 协议遵循度偏低 | 看门狗报告遵循度 < 60% | watchdog.py 输出 |

### 降级模式行为

```
检测到触发条件
  │
  ▼
输出 [Auto] ⏸ 降级: {原因}
  │
  ▼
切换策略:
  1. 当前未完成子任务标记为"保守模式"
  2. 后续子任务优先用 read_file + run_command，避免子代理
  3. Phase 5 交付报告中标注"降级运行"
  4. 对该项目在 state.json 标记"高风险"
  │
  ▼
继续执行（不中止）
```

### 集成步骤
1. 在 Phase 3 入口处（STEP 3.0 之后）插入判断
2. 检查 `state.json` 中是否有该项目的"高风险"标记
3. 读取 `routing_weights.json` 确认是否有技能的 weight_penalty ≥ 0.3
4. 从 `error_patterns.json` 统计近期同类错误频率
5. 条件满足时输出降级标记并切换策略

### 验收标准
- [ ] 连续失败时可观测到 `[Auto] ⏸ 降级` 标记
- [ ] 降级后不再派发子代理
- [ ] `state.json` 中有高风险标记写入

## W-4 动态重规划（P2 — 📋 设计就绪）

### 目的
长运行中，允许基于新信息重新规划未执行的任务，避免僵化执行过时计划。

### 触发时机

| 事件 | 重规划动作 |
|------|-----------|
| 新依赖被发现 | 插入新 Level+1 任务，更新依赖图 |
| 技能缺口出现 | 触发 write-a-skill，重新运行 Phase 1 装配 |
| 复杂度远超预估 | 拆分未开始的任务为更细粒度，重注入队列 |
| 外部依赖变更 | 动态调整优先级（阻塞类提高优先级）|

### 重规划流程

```
检测到重规划事件
  │
  ▼
输出 [Auto] 🔄 重规划: {原因}
  │
  ▼
暂停 Level 0 新派发（已在执行的继续）
  │
  ▼
更新依赖图:
  1. 新任务注入 → 按依赖关系分配 Level
  2. 阻塞任务标记 → 依赖解除后自动重试
  3. 优先级调整 → 关键路径优先
  │
  ▼
恢复派发
```

### 集成步骤
1. 在 Phase 2.3（ralph-planner 输出依赖图后）获取完整任务列表
2. 每个子任务完成后检查是否有新信息触发重规划
3. 重规划不修改已完成的子任务，只影响"待办"和"阻塞"队列

### 验收标准
- [ ] 新依赖被发现后，有新的子任务注入队列
- [ ] 已完成的子任务不被重规划修改

## H-1 可进化的任务拆分策略（P2 — 📋 设计就绪）

### 目的
通过记录每次拆分的准确度，逐步优化同类任务的拆分粒度。

### 数据收集

每次子任务完成后记录：

| 字段 | 来源 |
|------|------|
| 任务类型 | 路由表匹配的类型 |
| 预估耗时 | ralph-planner 输出 |
| 实际耗时 | 子任务完成时间 |
| 是否阻塞 | 是否标记为 blocked |
| 复杂度评分 | Phase 0 评估 |

### 调整规则

```
若 同一类型 的连续 5 次任务中:
  - ≥3 次实际耗时 > 预估 × 2 → 下次拆分粒度加倍（拆更细）
  - ≥3 次实际耗时 < 预估 × 0.5 → 下次合并相邻任务（拆更粗）
```

记录保持 50 条，超出后丢弃最旧。

### 验收标准
- [ ] 每个子任务完成后有耗时记录写入 `skill_performance.json`
- [ ] 同类任务连续高偏差后有可见的拆分粒度调整

## H-2 "后悔药"机制（事务级回滚）（P2 — 📋 设计就绪）

### 目的
高风险子任务执行前自动创建轻量级回滚点，失败时快速恢复。

### 实现方案

```
高风险任务（满足任一）:
  - 涉及 delete_file / write_file（覆盖）
  - 涉及数据库 schema 变更
  - 标记为 high 风险

执行前:
  1. 创建 _rollback/{task_id}/ 目录
  2. 备份被修改的文件原始内容
  3. 如 git 仓库已初始化 → git stash push -m "rollback-{task_id}"
  4. 记录回滚点到 state.json

执行后（成功）:
  删除 _rollback/{task_id}/ 目录

执行后（失败 ≥2 次）:
  1. 从 _rollback/{task_id}/ 恢复原始文件
  2. git stash pop（如适用）
  3. 输出 [Auto] ↩️ 回滚: {task_id} {原因}
  4. 标记任务为"高风险"
```

### 回滚点文件结构

```
.reasonix/_rollback/
  └── T-004/
      ├── manifest.json    # 记录备份了哪些文件、时间戳
      ├── src/
      │   └── main.py.bak
      └── config/
          └── settings.json.bak
```

### 集成步骤
1. Phase 3 STEP 3.1（拆分子任务时）识别高风险子任务
2. 高风险子任务派发前自动创建回滚点
3. ralph 循环中连续失败 2 次后触发恢复

### 验收标准
- [ ] 高风险任务派发前有 `_rollback/` 目录生成
- [ ] 失败 2 次后可观测到 `[Auto] ↩️ 回滚` 标记
- [ ] 回滚后文件内容与执行前一致

## H-3 隐性需求挖掘引擎（P2 — 📋 设计就绪）

### 目的
在 Phase 2 设计阶段主动发现可能被遗漏的需求，减少后期返工。

### 实现方案

在 Phase 2 brainstorming / grill 之后，追加隐性需求扫描：

```
1. 根据任务类型查隐性需求规则库
2. 规则库内容（持续积累）：

   web-frontend 类型:
     - 是否有 loading/empty/error 三态处理？
     - 是否有移动端响应式？
     - 是否有 SEO 基础（title/meta/og）？
     - 是否有 404 页面？
   
   backend-api 类型:
     - 是否有请求参数校验？
     - 是否有统一的错误响应格式？
     - 是否有 rate limiting？
     - 是否有健康检查端点？
   
   data-processing 类型:
     - 是否有空值 / 异常值处理？
     - 是否有大文件分块处理？
     - 是否有编码问题处理（UTF-8 vs GBK）？

3. 匹配命中的隐性需求 → 追加到任务列表
4. 每次后期补充的隐性需求自动追加到规则库
```

### 规则库存储

```json
// ~/.reasonix/hidden-needs-rules.json
{
  "web-frontend": [
    {"pattern": "loading|empty|error state", "question": "是否有 loading/empty/error 三态处理？"},
    {"pattern": "responsive", "question": "是否有移动端响应式布局？"}
  ],
  "backend-api": [
    {"pattern": "validation", "question": "是否有请求参数校验？"},
    {"pattern": "error format", "question": "是否有统一的错误响应格式？"}
  ]
}
```

### 集成步骤
1. Phase 2 brainstorming/grill 完成后执行
2. 根据任务类型加载对应规则
3. 每个匹配的问题追加为一个分析子任务
4. 用户在 Phase 2 结束时确认是否采纳

### 验收标准
- [ ] Phase 2 完成后输出隐性需求清单
- [ ] 用户确认后需求进入任务列表
- [ ] 新发现的隐性需求可追加到规则库

## 违例学习（P0 — ✅ 已在全流程实现）

### 目的
每次协议违例自动记录到 `error_patterns.json`，积累后自动优化流程。

### 当前实现
- 各 Phase 出口的违例检查已自动记录
- `STEP 3.6a` 和 `STEP 4.2.5` 强制记录新错误模式
- `routing_weights.json` 根据 confidence 自动调整权重
- `watchdog.py` 独立进程持续监控协议遵循度

### 格式

```json
{
  "ProtocolViolation": [
    {
      "timestamp": "2026-05-24 10:30:00",
      "count": 2,
      "details": ["L1:缺少 Auto 前缀", "L2:Phase 3 无子代理调用"],
      "severities": ["L1", "L2"],
      "skill_id": "full-autonomous",
      "phase": 3
    }
  ]
}
```

### 自动修复规则
同一模式累计 ≥2 次 → 下次执行时在对应 STEP 前自动输出 `⚠️ 历史违例高发步骤` 标记。
