# 扩展机制

> ⚠️ **DRAFT** — 以下机制均为设计草案，尚未实现。
> 不应在执行流程中引用这些机制。实现后将移除此标记并更新引用。

## G-4 技能清单快照

每次 Phase 0 启动时，将当前 `<available_skills>` 快照写入 `skill_snapshot.md`：

```
[技能快照] 会话启动于 {Y-m-d H:M}
可用技能: full-autonomous, diagnose, tdd, zoom-out, ...
```

用于后续会话校准和缺口检测。

## S-3 技能性能反馈回路

每次技能执行后记录：

```
[技能反馈] diagnose | 耗时: 45s | 结果: 成功 | 错误类型: 无
```

记录持久化至 `skill_performance.json`，供 skill-router 优化下次匹配排序。

## S-6 自动暂停决策模块

当检测到以下任一条件时，主动进入降级运行模式：
- 连续 3 个子任务耗时超过预估 2 倍
- 技能缺口率 > 30%（无法匹配 skill 的任务占比）
- 同一任务失败 3 次以上且原因不同
- 协议遵循度 < 60%（看门狗数据）

降级模式行为：
1. 对该项目标记"高风险"，未来同类任务优先用保守策略
2. 输出高优先级告警包含完整上下文
3. 不中止当前运行，但切换为"最小可行"策略

## W-4 动态重规划

在长运行中，允许基于新信息重新规划未执行的任务：
- 新依赖发现 → 更新路由表
- 技能缺口出现 → 触发 write-a-skill → 重新装配
- 复杂度远超预估 → 拆分任务更细粒度 → 重新注入队列
- 外部依赖突然可用/不可用 → 动态调整优先级

## H-1 可进化的任务拆分策略

记录每次拆分后：
- 实际 vs 预估耗时偏差
- 是否产生阻塞
- 复杂度是否误判

用于调整未来同类项目的拆分粒度。

## H-2 "后悔药"机制（事务级回滚）

高风险子任务执行前自动创建轻量级回滚点：
- 备份被修改的文件（`_rollback/`）
- 数据库 savepoint（如适用）
- git stash（如 git 仓库已初始化）

连续失败 > 2 次自动回滚并标记该任务为"高风险"。

## H-3 隐性需求挖掘引擎

在 Phase 2 设计阶段，主动扫描：
- 该项目类型的常见坑
- 类似项目的典型遗漏需求
- 每次后期补充的隐性需求自动记录

## 违例学习（violation learning）

每次协议违例自动记录到 `error_patterns.json`，格式：

```json
{
  "ProtocolViolation": [
    {
      "timestamp": "2026-05-21 10:30:00",
      "count": 2,
      "details": ["L1:缺少 Auto 前缀", "L2:Phase 3 无 task() 调用"],
      "phase": 3,
      "step": "3.2"
    }
  ]
}
```

当同一模式累计 ≥2 次 → 自动在对应 MANDATORY STEP 前添加 `⚠️ 历史违例高发` 标记。

## 退出条件

- 所有任务完成 → Phase 5 交付报告
- 所有剩余任务标记为阻塞 → 输出汇总，通知用户验收
- 用户发送"停"/"暂停"/"stop"打断
- 协议遵循度连续 3 次自检为 0% → 强制暂停，通知用户
