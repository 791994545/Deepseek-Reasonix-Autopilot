# 错误自愈与违例学习

## 错误处理流程

```
失败 → 输出 [Auto] 🔄 失败, 尝试方法B
  ├─ 方法 A 失败 → 试方法 B（完全不同思路）
  ├─ 方法 B 失败 → 试方法 C（查文档/搜索/替代方案）
  └─ 全部失败 → 记录阻塞点 → 继续其他不依赖子任务
```

## 回溯触发条件

| 触发条件 | 回溯目标 |
|----------|----------|
| 连续出现架构级错误 | 回到 Phase 2（设计阶段） |
| 集成测试发现需求误解 | 回到 Phase 0（重新分类） |
| 技能缺口导致阻塞 | 触发 write-a-skill → 继续 |
| 单元测试失败 | 当前子任务内 diagnose → 修复 |
| 协议偏离（自检发现） | 回到当前 MANDATORY STEP |
| Phase 门禁未通过 | 回到对应 Phase 重新执行门禁 |

## 回溯规则

- 回溯范围**局部化**，不丢弃已完成的无影响子任务
- 已通过的 Phase 无需全部重做，只重做涉及变更的部分
- 回溯后更新 checkpoint

## 违例自愈整合

### 5 层恢复栈

```
L1: 格式违规（缺少 [Auto] / 输出格式错误）
    → 即时纠正后续输出格式
    ↓ (未解决)
L2: 步骤缺失（跳过某 STEP 或 Phase 门禁）
    → 回溯到对应步骤重新执行
    ↓ (未解决)
L3: 策略僵化（同一方法连续失败 ≥3 次）
    → skill-router 降级到备选技能
    ↓ (未解决)
L4: 持续性偏离（连续 2 次自检发现同一问题）
    → 记录 error_patterns.json，强化指令
    ↓ (未解决)
L5: 完全失控（连续 3 次自检全部偏离）
    → 输出 `[协议违规] 无法自主恢复` → 通知用户
```

### diagnose 集成

```
diagnose 分析根因
  ├─ 代码错误 → 修复代码
  ├─ 架构问题 → 回溯 Phase 2
  ├─ 需求误解 → 回溯 Phase 0
  └─ 工具/SDK 问题 → 换方法/查文档
```

### self-improving 学习集成

每次违例恢复后：

1. 记录违例事件 → `error_patterns.json`
2. 如果同一模式发生 ≥2 次 → 对应 MANDATORY STEP 添加 `⚠️` 标记
3. 任务结束后 → 复盘输出包含违例统计

## 降级路由

当子任务首选技能失败时：
- skill-router 降级到备选技能
- 全候选失败 → 返回阻塞信号给编排器（不暂停整个流程）
- 阻塞任务不影响其他不依赖它的任务

## 关键规则

### 无中断原则
- 不 AskUserQuestion（除非外部依赖完全不可用）
- 不要求确认（删除/安装/重启等自动执行）
- 卡住了换策略，不跳过

### 时间/重试/成本上限
- 单子任务最大重试次数：3 次不同策略
- 单子任务最大耗时：30 分钟（超时标记阻塞，继续其他）
- 总任务耗时上限：启动时估算并声明

### 破坏性操作保护
任何破坏性操作（rm -rf、DROP TABLE、git push --force、付费 API）执行前：
1. 输出 `<danger>` 标签，说明理由、影响面和回滚方案
2. 无人值守下，若无回滚方案 → **强制暂停**
3. 日志记录操作详情
