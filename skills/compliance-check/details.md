# Compliance Check — 规则遵守审查

在关键节点自动调用，逐条对照规则评分。不符合项强制暂停修正。

---

## 调用时机

| 触发点 | 检查项 | 通过标准 |
|--------|--------|----------|
| Gate 1 完成 | 需求锚点确认 | 至少 1 个不可覆盖锚点已记录 |
| Gate 2 完成 | 安全设计审查 | security-best-practices 已执行 |
| 子任务开始前 | 技能预检 (Rule 0) | 当前任务有匹配技能 |
| 子任务开始前 | 读后才写 (Rule 8) | 目标文件已读取 |
| 代码提交前 | 简单性原则 (Rule 2) | 无未请求的抽象层 |
| Gate 3 完成 | 测试意图 (Rule 9) | 测试覆盖边界条件 |
| 交付前 | 全面审查 | 以上全部 + danger 日志完整 |

---

## 检查项详细定义

### 1. 技能预检 (Rule 0)
```
[合规检查] Rule 0 (技能预检)
- 用户请求中是否有技能触发词？→ 是/否
- 对应技能是否已加载？→ 是/否
- 若否 → ❌ 强制暂停，加载对应技能后继续
```

### 2. 简单性原则 (Rule 2)
```
[合规检查] Rule 2 (简单优先)
- 本次修改是否有未请求的抽象层？（接口/基类/工厂/配置化）→ 是/否
- 是否有未请求的框架引入？→ 是/否
- 代码行数是否显著超出问题域？→ 是/否
- 任意 "是" → ⚠️ 暂停修正
```

### 3. 读后才写 (Rule 8)
```
[合规检查] Rule 8 (读后才写)
- 本次修改涉及的每个文件是否已读取？→ 是/否
- 若否 → ❌ 强制暂停，先读取再修改
```

### 4. 测试意图 (Rule 9)
```
[合规检查] Rule 9 (测试意图)
- 业务逻辑变更时测试是否同步更新？→ 是/已通过
- 测试是否覆盖边界条件而非仅主路径？→ 是/否
- "否" → ⚠️ 暂停补充测试
```

### 5. 危险操作审计
```
[合规检查] 危险操作
- 本次操作是否包含破坏性命令？(rm -rf / DROP TABLE / force push / 付费API)
- 若包含:
  - 是否有 `<danger>` 标签？→ 是/否
  - 是否有回滚方案？→ 是/否
  - 缺任一 → ❌ 强制暂停
```

---

## 输出格式

每次检查结果输出结构化报告：

```
[合规检查] Gate 2 完成 — 安全设计
Rule 0 (技能预检): ✅
Rule 2 (简单优先): ⚠️ 发现未请求的抽象层: IRepository 接口
Rule 8 (读后才写): ✅
Rule 9 (测试意图): ✅
危险操作审计:   ✅ 无破坏性操作
─────────────────────────────────
判定: ⚠️ 存在警告项，修正后继续
修正建议: 移除 IRepository 接口，使用直接依赖
```

---

## 动作

| 判定 | 动作 |
|------|------|
| 全部 ✅ | 无操作，继续 |
| 存在 ⚠️ | 暂停当前任务，修正警告项后继续 |
| 存在 ❌ | 强制暂停，标记为硬阻塞，不可跳过 |
| 同一 ⚠️ 反复出现 2 次 | 升级为 ❌，记录为阻塞并切换备选策略 |

---

## 集成方式

由 `full-autonomous` 在 Gate 检查点自动加载。也可手动调用：

```
检查 Gate 2 合规性 → 自动加载 compliance-check 技能
```
