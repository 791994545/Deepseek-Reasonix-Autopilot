## Phase 0 — 初始化与分类

### 复杂度评估标准

| 维度 | 1分 | 2分 | 3分 |
|------|-----|-----|-----|
| 文件规模 | 1-3 文件 | 4-15 文件 | 16+ 文件 |
| 模块数量 | 1 模块 | 2-4 模块 | 5+ 模块 |
| 外部依赖 | 0 个 | 1-5 个 | 6+ 个 |
| API/服务集成 | 0 个 | 1-2 个 | 3+ 个 |
| UI 复杂度 | 无/纯逻辑 | 静态 UI | 交互式/多页面 |
| 数据持久化 | 无 | 本地文件 | 数据库/云存储 |

总分 2-4 → Quick, 5-9 → Standard, 10-18 → Full

### 任务类型匹配

skill-router 根据需求描述中的关键词匹配任务类型（见 `rules/01-skill-routing-table.md`）：

```
"帮我做一个 landing page" → web-frontend
"写一个 MCP 服务器" → backend-api
"全栈电商应用" → web-fullstack
"自动化浏览器操作" → browser-auto
```

## Phase 1 — 动态技能装配

### 技能包筛选规则

```
技能包 = Core(T) + (路径==Standard|Full ? Enhanced(T) : []) + (路径==Full ? Extended(T) : []) + (路径!=Quick ? Testing(T) : [])
```

其中 T = 任务类型。

## Phase 3 — 自适应执行

### 子代理派发

```
Level 0 (无依赖)  → 同一消息中 N 个并行 task()
Level 1 (依赖 L0) → 等待 L0 聚合后派发
Level 2 (依赖 L1) → 等待 L1 聚合后派发
...
```

### 每个子任务的执行循环

```
循环直到通过:
  1. 写测试（如适用）
  2. 实现代码
  3. 运行测试
  4. 通过 → 完成
  5. 失败 → diagnose → 换策略 → 重试
  6. 3次不同策略仍失败 → 标记阻塞 → 继续其他
```

## Phase 4 — 自适应验证

验证技能根据 Phase 1 加载的技能组合包自动决定：

```
技能包含 web-dev → 执行 webapp-testing
技能包含 frontend-design → 执行 web-design-guidelines
技能包含 midscene-test → 执行 UI 自动化测试
Full 路径 → 强制 security-best-practices 最终审查
```

## Phase 5 — 交付与学习

### 交付报告内容

```
完成: N 个子任务
阻塞: M 个子任务（原因）
跳过: K 个子任务（原因）
总耗时: Xm
协议遵循度: P% (N偏离 / 总检查点)
技能利用率: L% (N技能使用 / N技能加载)
学习记录: self-improving 复盘输出
```

### 协议遵循度计算

```
协议遵循度 = (1 - 偏离次数 / 总检查点数) × 100%
检查点 = MANDATORY 步骤数 + Phase 门禁项数
```
